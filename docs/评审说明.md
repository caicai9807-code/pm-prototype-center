# 评审说明

## 一、评审目标

本次评审围绕「面积检查与整改业务」展开，通过可交互的 HTML 原型确认以下内容：

1. 业务流程是否完整、合理
2. 页面结构和字段是否满足业务需求
3. 各角色操作权限和流转规则是否正确
4. 异常流程和边界情况是否已覆盖
5. 字段口径和枚举值是否准确
6. 各端（PC / 专家 App / 部门领导 App）职责边界是否清晰

## 三、非本次评审范围

- 真实后端接口设计（本原型使用 Mock 数据）
- 数据库表结构设计（开发阶段另行输出）
- 技术架构和部署方案
- 性能、安全、高可用等非功能需求
- 与外部系统的数据对接方案
- UI 视觉细节（字体、间距、色彩精确值）

## 四、推荐评审路径

请按以下流程操作原型，逐环节确认：

1. **业务管理员创建面积检查任务**
   - 进入 task-list.html → 点击「+ 新建任务」
   - 填写基础信息 → 选择站点
   - 进入专家抽取 → 分配专家 → 提交任务

2. **专家确认或拒绝**
   - 切换专家视角 → 进入 my-task-list.html
   - 确认任务（待确认→待开始）
   - 或拒绝任务（触发转派流程）

3. **部门领导转派（如专家拒绝）**
   - 进入部门领导 App → 审核指派
   - 选择新专家 → 确认转派

4. **专家开始检查**
   - 进入 my-task-check.html
   - 查看普查信息
   - 填写检查记录单
   - 不合格时生成整改通知单

5. **双人确认**
   - 检查记录单提交后进入确认中状态
   - 两位检查人各自同意或驳回
   - 全部同意后站点完成

6. **整改通知书审核**
   - 进入 review-notice.html
   - 审核整改通知书（通过/驳回）

7. **整改任务流转**
   - 整改责任人接收整改任务
   - 填写整改说明 → 提交整改
   - 双人复查确认 → 整改完成归档

## 五、角色说明

| 角色 | 职责 | 主要操作 |
| --- | --- | --- |
| 业务管理员 | 创建和管理检查任务 | 新建/编辑任务、分配专家、审核整改通知书 |
| 检查专家 | 执行现场检查 | 确认任务、填写检查记录单、参与确认 |
| 部门领导 | 处理专家转派 | 审核拒绝原因、指派新专家 |
| 整改责任人 | 执行整改 | 接收任务、提交整改说明和附件 |
| 管理部主任 | 查看本部门情况 | 查看统计、不直接参与操作 |

## 六、业务主流程

```
业务管理员创建任务 → 选择站点 → 抽选分配专家 → 专家确认任务
  → 专家开始检查 → 填写检查记录单
    → 合格 → 站点完成
    → 不合格 → 生成整改通知书 → 双人确认
      → 审核整改通知书 → 生成整改任务
        → 整改责任人接收 → 提交整改
          → 双人复查确认 → 整改完成归档
```

## 七、重点确认事项

1. **任务创建字段是否完整**：任务名称、年度、周期、站点选择等字段是否符合业务要求。
2. **专家抽选规则是否符合业务**：年龄优先、检查间隔、管理部专家配置、抽取数量。
3. **专家拒绝后是否必须走部门领导转派**：拒绝后的流转路径和界面提示。
4. **检查记录单字段和结论生成规则是否正确**：7项检查项是否符合实际检查标准。
5. **普查依据为"实测"的建筑是否需要复核面积**：实测复核 Tab 的展示规则。
6. **实测复核误差规则是否确认**：三档误差阈值（≤500㎡ / 500~2000㎡ / >2000㎡）。
7. **整改通知书是否需要双人确认**：确认流程和状态展示。
8. **整改期限超过 30 工作日是否需要业务管理员审核**：≤30日自动通过，>30日人工审核。
9. **整改任务负责人是否按站点所属片区所所长自动生成**：负责人映射规则。
10. **整改完成是否必须双人复查确认**：复查流程和驳回处理。
