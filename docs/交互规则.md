# 交互规则

用于定义页面中的校验、联动、状态切换、权限差异和异常提示规则。

## 建议填写规则

- 优先记录会影响演示效果和业务理解的关键交互。
- 一个规则描述一个明确场景，避免混写。
- 对按钮可用性、字段显隐、审批动作、批量操作要写清前置条件。
- 若规则依赖角色或状态，必须显式写出。

## 重点关注

- 查询筛选联动：各列表页筛选条件变更后需点击「查询」才生效，「重置」清空所有条件
- 表单校验：task-create.html 下一步前校验3个必填字段+站点选择；task-expert.html 提交前校验抽取状态+槽位分配；my-task-list.html 拒绝弹窗校验拒绝原因必填；my-task-check.html 提交前校验检查结论+检查项+不合格原因
- 按钮显隐和可用态：按任务状态控制编辑/删除链接可用性；按选择状态控制弹窗确认按钮可用性；my-task-list.html 按任务状态显示不同操作按钮组合；my-task-check.html 检查项select联动控制原因input可用性
- 状态流转：面积检查任务（草稿→待开始→进行中→已结束）；专家视角新增「待确认」状态（确认→待开始，拒绝→退回专家所属部门领导重新分配）；站点检查状态（待检查→检查中→已检查/不合格）
- 批量操作：站点选择弹窗支持全选 checkbox
- 弹窗/抽屉反馈：站点选择弹窗、专家选择弹窗、拒绝任务弹窗均支持遮罩层点击关闭
- 空状态、异常状态、无权限状态：见 7.3 空状态规则
- 整改任务流程（新增）：待整改 →（接收确认，开始计时）→ 整改中 →（超期派生为「已超期」，仍可整改）→（保存整改信息）→ 待专家审核 →（双专家会签：专家A 张明远 + 专家B 李建华，均同意）→ 已完成；任一专家不同意 → 退回整改中。详见 §八、§九
- 整改通知书审核列表「详情」改为跳转站点详情页（新标签页），「审核」保留为弹窗（仅待审核可见）；详见 §七 7.2
- 编辑模式交互：task-list 编辑按钮→task-create.html?id={id}&mode=edit→task-expert.html?id={id}&mode=edit，复用新建两步流程
- 编辑范围按状态控制：草稿/待开始全可编辑，进行中仅可编辑专家分配（基础信息+站点锁定），已结束不可编辑
- 跨页面传参：URL 参数（id + mode=edit）为主通道，sessionStorage（editMode + editTaskId）为 fallback；my-task-check.html 通过 URL 参数 id={taskId} 接收任务ID
- 操作列固定：my-task-list.html 操作列 sticky 固定右侧，表格横向滚动时操作列始终可见
- 站点Tab切换联动：my-task-check.html 站点Tab切换时动态重渲染并重新绑定子Tab/详情弹窗/检查记录单/通知单/确认/复核事件
- 检查记录单弹窗：7项检查项radio+checkbox子项联动，选"否"展开子项，选"是/不适用"隐藏并清空
- 双专家确认：检查记录单和整改通知单展示态均支持第一专家→第二专家→已生效的确认流程
- 动态Tab系统：结论合格时隐藏整改通知单Tab；普查依据含"实测"时显示实测复核Tab
- 实测复核Tab：过滤普查依据为"实测"的建筑，照片上传+复查面积输入+保存校验
